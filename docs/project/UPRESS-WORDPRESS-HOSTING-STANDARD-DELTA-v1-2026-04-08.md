## 2. עקרונות שעיגנו את ההחלטות

- **נעילת ספק:** uPress כברירת מחדל ארגונית, עם קישור לתיעוד הרשמי שלהם.  
- **פורט FTP 21** כתקן ברירת מחדל.  
- **הבחנה סטייג'ינג / פרודקשן:** ייתכן שתצורת SSL/TLS ו־FTP תשתנה בין סביבות; אין להסיק ממקרה אחד בלי בדיקה; **לאמץ את הגישה המחמירה** ככל שניתן.  
- **אבטחה וסודות:** הקפדה מלאה; אחידות בתבניות — בלי הגזמה בפיצול פורמטים בין פרויקטים.  
- **דוקר:** **חובה** במודל המלא — חד־פעמי, יתרון ארוך טווח; נוהל מחייב בפרויקטים.  
- **תוספים:** סטנדרט והמלצה — **לא חובה** זהה לכל אתר; תבנית שונה בין אתרים — לא להשוות «תוסף מול תוסף» כרלוונטיות גלובלית.  
- **פיתוח מול פרודקשן:** בפיתוח — **מירב יכולות** לסוכנים (REST, אוטומציה, בדיקות); לאחר **נעילת סבב** ופריסה סופית לפרודקשן — **הקשחה** והגבלת אופציות מקוונות.  
- **מיקוד מסמך הנוהל:** רק **WordPress + uPress + כלי גישה**; לא Cursor/צוותים/שערי בקרה פנימיים — אלה נשארים במסמכי ניהול אחרים.

## 4. מה מבקשים מצוות העמית (משוב)

1. האם **תיאור מגבלות uPress** (אין SSH, אין MySQL מרחוק) תואם את הניסיון שלכם בפרויקטים נוספים?  
2. האם יש **תכונות פאנל** (גרסה חדשה, API, CLI) שכדאי להוסיף לנוהל v1.1?  
3. **FTPS / session reuse** — האם נתקלתם בהבדלים מדויקים בין סטייג'ינג לפרודקשן על אותו ספק?  
4. האם **רשימת תוספים סטנדרטית** מוצעת מכם תתאים לסנכרון מול ספריית uPress?  
5. הערות כלליות על **§9 Hub** כתבנית ללקוחות אחרים.

## 5. עקביות מול מסמך ההשוואה הטבלאי

מסמך הסינטזה הטבלאי נשאר **בסיס ראיות**; הנוהל v1 **מצמצם** לתקנים שאושרו מנהלית ו**מסיר** נושאים שהוחרגו במפורש (צוותים, Cursor, שערים). סעיפים טכניים מ־Bentiv (TLS reuse, REST, phpMyAdmin) **נבלעו** לנוהל רק כאשר הופיעו בהחלטות בסעיף 3 לעיל.
