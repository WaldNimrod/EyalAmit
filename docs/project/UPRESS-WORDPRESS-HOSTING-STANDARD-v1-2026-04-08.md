## 2. FTP — תקן ברירת מחדל

| פרמטר | נעילה |
|--------|--------|
| **פורט** | **21** — ברירת המחדל הארגונית לכל חיבורי FTP ל־uPress. |
| **יצירת חשבון / נתיב** | לפי תיעוד uPress: [יצירת חשבון FTP](https://support.upress.co.il/dev/create-new-ftp/) · [שימוש ב־FTP](https://support.upress.co.il/dev/how-to-use-ftp/). |
| **TLS / FTPS** | יש **לבחון פר־סביבה** (סטייג'ינג מול פרודקשן): ייתכן הבדל בהגדרות הצפנה בין סביבת פיתוח לבין אתר חי. **חובה** לאמץ את הגישה **המחמירה ביותר** שמתאימה לשרת בפועל (כולל FTPS עם session reuse על נתונים אם נדרש — כמפורט בדוח השוואתי מול Playbook חיצוני). |
| **HTTPS בדפדפן (סטייג'ינג)** | על hostname של **sandbox** לרוב **אין** תעודת SSL ציבורית תקינה; לבדיקות פונקציונליות (כולל Hub סטטי) השתמשו ב־**`http://`**. **בפרודקשן** — רק **`https://`** עם תעודה מותקנת. ראו [`UPRESS_WORDPRESS_STANDARD_v2.md`](./UPRESS_WORDPRESS_STANDARD_v2.md) §2.1. |
| **סקריפטים** | פרויקטים ישתמשו בסקריפטים קנוניים עם `dry-run`, נתיב מרוחק מפורש, ודילוג על שמות קבצים/נתיבים לא חוקיים. |

## 3. חשבונות FTP — סטנדרט ודריסה

| כלל | פירוט |
|-----|--------|
| **ברירת מחדל** | חשבון FTP אחד **סטנדרטי** לפרויקט: גישה לשורש WordPress של אותו אתר (או לנתיב המוגדר ב־uPress כברירת מחדל לפי התיעוד), פורט 21, שמירת פרטים ב־תבנית credentials מאוחדת (ראו דלטה). |
| **דריסה לפי פרויקט** | מותר להגדיר חשבונות נוספים או נתיבים מצומצמים (למשל תיקייה סטטית נפרדת) כאשר הפרויקט דורש — **חובה לתעד** ב־runbook / נספח פרויקט: מטרה, נתיב, מי מחזיק בסיסמה. |
| **הפרדת hub סטטי** | כאשר קיים ממשק סטטי מחוץ ל־WP (ראו §9), מותר נתיב FTP נפרד יחסית לשורש החשבון — כפי שממומש בפרויקט ייחוס. |

## 4. phpMyAdmin ועבודה מול מסד הנתונים

### 4.1 מה קיים ב־uPress

לפי התיעוד הרשמי: גישה ל־**phpMyAdmin** דרך הפאנל (לשונית פיתוח → ניהול מסדי נתונים → התחברות לממשק). מדריכים קנוניים:

- [התחברות ל־phpMyAdmin](https://support.upress.co.il/dev/how-to-login-myadmin/)
- [פרטי התחברות למסד](https://support.upress.co.il/dev/obtaining-database-details/)
- [יצירת מסד נתונים](https://support.upress.co.il/dev/create-new-database/)

### 4.2 מגבלות פלטפורמה (רלוונטי לאוטומציה)

- **MySQL מרחוק (פורט 3306)** אינו זמין מחוץ לשרת — אין חיבור ישיר מכלי CLI מקומי למסד הפרודקשן/סטייג'ינג בלי ממשק ביניים.
- **אין SSH סטנדרטי** — אין «נקודת הרצת סקריפטים» על השרת בסגנון shell או WP-CLI על השרת, כפי שמקובל ב־VPS.

### 4.3 מה חסר לנו לאיכות עבודה (ושיפורים ארגוניים)

| פער | השלמה מומלצת |
|-----|----------------|
| **אוטומציה headless ל־DB** | אין תחליף מלא ל־SSH; יש לשלב: **REST API + Application Password** לפעולות שאפשר דרך WordPress, **WP-CLI מקומי בדוקר** על עותק/ייצוא, ו־**phpMyAdmin** לשאילתות / תיקונים נקודתיים. |
| **תיעוד אחיד per אתר** | לכל אתר: URL מלא ל־phpMyAdmin, `DB_NAME`, משתמש, **קידומת טבלאות** — בשדות קבועים בקובץ credentials מקומי (לא ב־Git). |
| **ספר שאילתות מאושרות** | תבנית לשאילתות ביקורת (אפשרויות פעילות, טבלאות קריטיות) — מחוץ למסמך זה ניתן לשמור ב־runbook צוות תשתית. |
| **מניעת שחיתות serialized** | איסור עדכון `REPLACE` גולמי על עמודות עם נתונים מסולסלים; שימוש ב־`get_option` / `update_option` או ביצוע דרך PHP חד־פעמי מבוקר. |
| **ביקורת סיכונים לפני SQL ישיר** | לכל פרויקט: רשימת מקרים שבהם **מותר** נוהל מסוכן (SQL ישיר, hooks חד־פעמיים) — **חובה בחינה** ואישור; ברירת המחדל — נמוך סיכון בלבד. |

### 4.4 «נקודה נוחה להרצת סקריפטים על השרת»

**לפי תיעוד uPress:** אין מתן shell או הרצת סקריפטים כלליים על השרת. הנקודות הלגיטימיות הן:

1. **מנוע WordPress** — קוד PHP בתוסף / תמה / mu-plugin (כולל hooks חד־פעמיים מבוקרים).  
2. **REST API** — אוטומציה עם הרשאות מתאימות.  
3. **phpMyAdmin** — SQL ידני/מבוקר.  
4. **WP-Cron / תוספים** — לפי צורך, בתיעוד הפרויקט.

**מסקנה ארגונית:** «נוחות סקריפטים» מושגת ב־**דוקר מקומי + סקריפטי פריסה**, לא ב־נקודת SSH על uPress. זה **מצב תקין** לפלטפורמה; השיפור הוא **בתהליכים ובתבניות**, לא בציפייה ל־shell על השרת.

## 5. סביבות: סטייג'ינג מול פרודקשן

לפי תיעוד uPress על **סביבות פיתוח / sandbox**: [סביבות פיתוח](https://www.upress.co.il/features/sandboxes/) · [ייבוא לסביבת פיתוח](https://support.upress.co.il/dev/import-to-sandbox/).

| כלל | פירוט |
|-----|--------|
| **הבדלי TLS/FTP** | יש להניח שסטייג'ינג ופרודקשן עשויים להיות מוגדרים אחרת; **לא לכלול** מסקנה מסביבה אחת בלי אימות בשנייה. |
| **הקשחה בפרודקשן** | בשלב **נעילת סבב** ומעבר לפרודקשן סופי: **הקשחה מחמירה** — צמצום משטח התקיפה, ביטול או הגבלת אמצעי «פיתוח מקוון» שאינם נדרשים, לפי תכנית הפרויקט. |
| **מצב פיתוח פעיל** | כל עוד הפרויקט בפיתוח: יש **לאפשר** לסוכנים ולכלים יכולות מיטביות (REST, בדיקות, אוטומציה) תוך שמירה על סודות ועל נוהל אבטחה — ראו §7. |

## 9. ממשק לקוח סטטי (Hub) — המלצה ארגונית

**לאמץ** את המודל שפותח לפרויקט Eyal Amit כ**תבנית המלצה** לממשק תקשורת מובנה מול לקוח (מחוץ ל־wp-admin), כאשר רלוונטי:

| רכיב | הפניה במאגר Eyal Amit 2026 |
|------|------------------------------|
| מיקום מקור | `hub/` (שורש המאגר) |
| בנייה | `scripts/build_eyal_client_hub.py` (אופציה `--mirror-docs` / `--mirror-docx`) |
| פריסה ל־FTP | `scripts/ftp_publish_eyal_client_hub.py` |
| נוהל SSOT משוב | `hub/EYAL-HUB-SSOT-WORKFLOW.md` |
| קליטת JSON | `scripts/ingest_eyal_feedback_json.py` |
| סקיל Cursor (פרויקטלי) | `.cursor/skills/eyal-client-hub-publish/SKILL.md` |

עקרון: תוכן ב־`data/*.json`, פלט סטטי ב־`dist/`, נתיב FTP נפרד (למשל `ea-eyal-hub`), ללא חובת PHP בשרת (אופציונלי `incoming/`).

## 10. סיכום החלטות קצר

1. **uPress** = ברירת מחדל ארגונית + קישורי תיעוד קנוניים.  
2. **FTP פורט 21** = ברירת מחדל; TLS לפי סביבה, גישה **מחמירה**.  
3. **חשבונות FTP** = סטנדרט + דריסה מתועדת לפרויקט.  
4. **DB** = phpMyAdmin מלא כערוץ רשמי; השלמת פערים באוטומציה דרך REST + דוקר + תיעוד; אין shell לסקריפטים על השרת — זה תואם uPress.  
5. **דוקר** = חובה במודל המלא.  
6. **תוספים** = סטנדרטизаציה מול ספריית uPress כרשימת המלצות, לא חובה אחידה לכל אתר.  
7. **REST** = מירב יכולות בפיתוח; הקשחה בפרודקשן אחרי נעילה.  
8. **Hub** = לאמץ כתבנית עם הרפרנסים לעיל.
